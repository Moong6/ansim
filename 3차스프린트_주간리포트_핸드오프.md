# 케어알림장 — 3차 스프린트 핸드오프 (주간 안심 리포트) v1.0

> **이 문서의 성격**: 1차(MVP)·2차(다국어)가 **이미 구현 완료된 상태**에 얹는 **델타 문서**입니다. 1차 산출물(`개발_핸드오프_문서_v3.md`, `init_db.sql`, 1차 Playbook)과 2차 산출물(`2차스프린트_다국어_핸드오프.md`)을 먼저 읽은 전제로 작성되었습니다.
>
> **3차 스프린트 범위**: 주간 안심 리포트 **하나만**. 한 주(월~일)간 쌓인 알림장 데이터를 AI가 분석해 종합 편지로 생성·발송. (보호자 채널, 스마트 문의, 면회 신청 등은 4차 이후)
>
> **변경 총량**: 신규 테이블 1개(`report`) + 신규 API 3개(`/api/reports/*`) + 신규 화면 1개(`/reports`) + 대시보드 헤더에 진입 링크 1줄. 기존 1·2차 API와 화면은 **전혀 건드리지 않음**.

## 1단계. 요구사항 (확정)

| 항목 | 내용 |
|---|---|
| 기능명 | 주간 안심 리포트 |
| 한 줄 정의 | 한 주(월~일)간 쌓인 어르신의 알림장 데이터를 AI가 분석해, 주의 흐름과 변화를 담은 따뜻한 종합 편지로 자동 생성 |
| 타겟 사용자 | 작성: 요양보호사·사회복지사 / 수혜: 보호자 |
| 해결 문제 | 매일의 알림장은 단편적이라 "이번 주 어르신이 전반적으로 어떠셨는지" 큰 그림이 안 보임. 직원이 한 주를 직접 요약하기엔 부담 |
| 핵심 동작 | ① 어르신·주 선택 → ② 서버가 해당 주 알림장 집계 → ③ AI가 흐름 분석해 편지 생성 → ④ 직원 검수·수정 → ⑤ 전송 |
| 데이터 입력원 | **사람 메모 아님.** 해당 주 notice 테이블의 `structured_status`, `participated_programs`, `final_polished_text` |
| 주 정의 | **월요일~일요일** (ISO 8601). 기본 선택 = 지난주 |
| 범위 안 | 어르신 1명 + 특정 주 → 리포트 1건. 통계 요약(코드 집계) + AI 편지 |
| 범위 밖 | 일괄 생성, 차트·그래프 시각화, PDF 내보내기, 보호자 채널, 자동 스케줄링, 버전 관리 |

### 1·2차 자산 재활용
새로 필요한 게 거의 없음. 데이터는 이미 notice 테이블에 쌓여 있고(append-only로 이력 온전), 톤 enum·전송 패턴·precautions 주입·검수 textarea는 1차 그대로. 새로 만드는 건 "한 주 집계 + 종합 편지 생성" AI 엔드포인트가 핵심.

## 2단계. 사용자 흐름 (요약)

```
[대시보드] 헤더에 [📊 주간 리포트] 링크 추가
      │
      ▼
[/reports] 주간 리포트 화면
 [1] 어르신 선택  [2] 주 선택(기본: 지난주)  [3] [리포트 생성]
      │
      ▼
[서버] preview API: 해당 주 데이터 집계 + 기록 일수 확인 (AI 호출 X)
      │
      ├ 0일      → 생성 차단 "기록이 없어 만들 수 없습니다"
      ├ 1~2일    → 경고 모달 "N일치뿐, 계속할까요?" → 확인 시 진행
      └ 3일+     → 정상 진행
      │
      ▼
[화면] R2 통계 요약(즉시 표시) + R3 AI 편지 로딩
      │
[서버] generate API: 통계+일별 본문+precautions를 프롬프트로 → Gemini → 단일 편지
      │
      ▼
[R3] AI 편지 (편집 가능 textarea) + [R4] 톤·전송
      │
 [전송하기] → send API → report INSERT + 발송
```

### 데이터 부족 분기 (할루시네이션 방어 핵심)
| 기록 일수 | 동작 |
|---|---|
| 0일 | 차단 안내, AI 호출 안 함 |
| 1~2일 | confirm 경고 후 진행. 프롬프트에 "N일치 기준, 없는 날 추측 금지" 명시 |
| 3일+ | 정상 생성 |

핵심은 **AI에게 "한 주"라고 뭉뚱그려 주지 않고 "며칠치"인지 명시**해 거짓 요약 차단. 1차 precautions 주입과 같은 철학.

## 3단계. 화면 명세

### 변경: 대시보드 헤더
1차 헤더에 `[📊 주간 리포트]` 링크 1줄 추가. `/reports`로 이동. 그 외 대시보드 변경 없음.

### 신규: /reports 화면 (R1~R4)

#### R1. 선택 영역 (항상 표시)
| 요소 | 내용 |
|---|---|
| 제목 | "📊 주간 안심 리포트" |
| 어르신 선택 | 담당 어르신 중 1명 (드롭다운 또는 카드) |
| 주 선택 | [지난주 5/18~5/24] [이번 주 5/25~5/31] [직접 선택], 기본=지난주 |
| 직접 선택 | 월요일만 시작점으로 (선택적) |
| [리포트 생성] | 어르신+주 선택 시 활성 |

#### R1 분기 (생성 클릭 후 preview 응답에 따라)
- `dataLevel: NONE` → 안내 박스, R2~R4 미표시, generate 호출 안 함
- `dataLevel: SPARSE` → `confirm` 모달 → 확인 시 R2 표시 + generate 호출
- `dataLevel: SUFFICIENT` → 바로 R2 표시 + generate 호출

#### R2. 통계 요약 (코드 집계, 읽기 전용)
| 요소 | 내용 예 |
|---|---|
| 제목 | "이번 주 기록 요약 (5/18~5/24)" |
| 기록 일수 | "기록된 날: 5일 / 7일" |
| 식사 분포 | "완식 3일 · 보통 2일" |
| 기분 분포 | "좋음 4일 · 보통 1일" |
| 건강 분포 | "좋음 5일" |
| 참여 활동 | "가장 많이 참여: 인지치료 4회, 실버체조 2회" |

차트 X, 텍스트/간단 막대만. preview 응답으로 즉시 렌더.

#### R3. AI 종합 편지 (생성, 편집 가능)
| 요소 | 내용 |
|---|---|
| 제목 | "주간 종합 편지" |
| 로딩 | "🤖 한 주의 기록을 모아 편지를 작성 중입니다..." |
| 본문 | 편집 가능 textarea (12줄+) — **단일 생성**, 3안 비교 없음 |
| 글자수 | 우측 하단 |

3단계에서 3안 → 단일로 결정(주간 편지는 길어 3개 비교 부담). 1차 3안 UI 컴포넌트를 `mode="single"`로 재사용 가능.

#### R4. 톤 + 전송 (1차 패턴 재사용)
| 요소 | 내용 |
|---|---|
| 톤 선택 | [친근] [정중(기본)] [공감·위로] |
| [🔄 다시 생성] | 같은 데이터로 편지만 새로 (선택적, 1회) |
| [전송하기] | Double Submit 방지(1차 방어 3) → 저장+발송 |
| 전송 후 | 토스트 "주간 리포트가 전송되었습니다" |

### 전송 정책
R2 통계 raw 숫자는 **직원 검수용으로만** 표시. 보호자에겐 R3 편지 본문에 자연스럽게 녹여 전송.

## 4단계. 데이터 모델 (델타)

### 신규 테이블: report
notice와 분리(데이터 구조가 다름 — notice는 일간 단일값, report는 주간 분포). 1차의 `notice_tone`, `notice_status` enum 재사용.

| 컬럼 | 타입 | 제약 | 설명 |
|---|---|---|---|
| id | bigint | PK | |
| resident_id | bigint | FK → resident | 대상 어르신 |
| author_id | bigint | FK → app_user | 생성 직원 |
| period_start | date | NOT NULL | 주 시작일(월요일) |
| period_end | date | NOT NULL | 주 종료일(일요일) |
| recorded_days | smallint | NOT NULL | 실제 기록 일수(0~7) |
| stats_summary | jsonb | NOT NULL | 코드 집계 통계 |
| source_notice_ids | jsonb | NOT NULL default '[]' | 근거 notice id 배열 (추적·검증용) |
| tone | notice_tone | NOT NULL default 'POLITE' | 1차 enum 재사용 |
| ai_generated_text | text | | AI 원본 |
| final_text | text | | 직원 편집 최종본 (UPDATE 금지) |
| status | notice_status | NOT NULL default 'DRAFT' | 1차 enum 재사용 |
| sent_at | timestamptz | NULL | |
| read_at | timestamptz | NULL | 보호자 읽음 시각 |
| created_at / updated_at / deleted_at | timestamptz | | soft delete |

### stats_summary JSONB 스키마
```json
{
  "recordedDays": 5,
  "meal":   { "FULL": 3, "NORMAL": 2, "LITTLE": 0, "REFUSED": 0 },
  "mood":   { "GOOD": 4, "NORMAL": 1, "ANXIOUS": 0 },
  "health": { "GOOD": 5, "NORMAL": 0, "NEEDS_OBSERVATION": 0 },
  "topPrograms": [ { "title": "인지치료", "count": 4 } ]
}
```

### source_notice_ids
근거 notice id 배열. AI가 뭘 보고 썼는지 추적 — 1차의 "엔진 검증 로그" 철학 연장.
```json
[101, 105, 112, 118, 124]
```

### 마이그레이션 DDL
```sql
-- 3차 스프린트: 주간 안심 리포트
CREATE TABLE report (
    id                BIGSERIAL PRIMARY KEY,
    resident_id       BIGINT NOT NULL REFERENCES resident(id),
    author_id         BIGINT NOT NULL REFERENCES app_user(id),
    period_start      DATE NOT NULL,
    period_end        DATE NOT NULL,
    recorded_days     SMALLINT NOT NULL,
    stats_summary     JSONB NOT NULL,
    source_notice_ids JSONB NOT NULL DEFAULT '[]',
    tone              notice_tone NOT NULL DEFAULT 'POLITE',
    ai_generated_text TEXT,
    final_text        TEXT,
    status            notice_status NOT NULL DEFAULT 'DRAFT',
    sent_at           TIMESTAMPTZ,
    read_at           TIMESTAMPTZ,
    created_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
    deleted_at        TIMESTAMPTZ
);
CREATE INDEX idx_report_resident_period
    ON report (resident_id, period_start) WHERE deleted_at IS NULL;
```
운영 중 DB에 `CREATE TABLE`만 추가. 기존 테이블 영향 없음. enum은 1차 것을 재사용하므로 추가 X.

### 시드 (데모용)
김순자(resident_id=1)의 **지난주 notice 5건**을 날짜 분산해 INSERT(없으면 SPARSE/NONE 분기 테스트). report 행은 시드 불필요(생성 데모로 만들면 됨).

### report에는 버전 관리 없음
1차 notice는 append-only(version)지만 report는 단순 INSERT. 수정 시 그냥 새 리포트 생성(빈도 낮음, MVP에 과함).

## 5단계. API 명세

### 신규 엔드포인트 요약
| # | 메서드 | 경로 | 역할 | 인증 |
|---|---|---|---|---|
| 1 | GET | /api/reports/preview | 주 데이터 집계 + 기록 일수 (AI 호출 X) | ✓ |
| 2 | POST | /api/reports/generate | AI 편지 생성 | ✓ |
| 3 | POST | /api/reports/send | report INSERT + 발송 | ✓ |

### preview와 generate를 분리한 이유
부족 분기 때문. 0일이면 AI 안 부르고 차단, 1~2일이면 사용자 확인 뒤 호출. 합치면 0일에도 AI 부르는 낭비 발생. 분리하면 preview는 통계만(LLM 비용 0), 사용자 결정 후 generate.

### 1. GET /api/reports/preview
**요청**: `?residentId=1&periodStart=2026-05-18` (periodStart는 월요일 날짜 필수)

**응답 200**
```json
{
  "residentId": 1,
  "periodStart": "2026-05-18", "periodEnd": "2026-05-24",
  "recordedDays": 5,
  "dataLevel": "SUFFICIENT",
  "statsSummary": { "recordedDays":5, "meal":{"FULL":3,"NORMAL":2,"LITTLE":0,"REFUSED":0},
    "mood":{}, "health":{}, "topPrograms":[{"title":"인지치료","count":4}] },
  "sourceNoticeIds": [101,105,112,118,124]
}
```
`dataLevel`: `NONE`(0) / `SPARSE`(1~2) / `SUFFICIENT`(3+). 프론트 분기 핵심.
**에러**: `400 INVALID_PERIOD`(월요일 아님) · `401` · `403 FORBIDDEN` · `404 RESIDENT_NOT_FOUND`

### 2. POST /api/reports/generate ⭐
**요청**: `{ "residentId":1, "periodStart":"2026-05-18", "tone":"POLITE" }`

**응답 200**
```json
{
  "periodStart":"2026-05-18", "periodEnd":"2026-05-24", "recordedDays":5,
  "reportText":"보호자님, 지난 한 주 동안 김순자 어르신께서는...",
  "statsSummary": {}, "sourceNoticeIds":[101,105,112,118,124]
}
```

**서버 로직 (필수 순서)**:
1. 해당 주 notice 조회 (status=SENT, root 그룹의 MAX version)
2. **0일이면 422 NO_DATA로 거부 (AI 호출 안 함)** — 프론트가 막아도 서버에서 다시 막음
3. structured_status 집계 → 통계 (코드, AI 아님)
4. 일별 final_polished_text + 통계 + **precautions** + 톤을 프롬프트로 조립
5. **프롬프트에 "기록 N일치 기준, 없는 날 추측 금지" 명시**
6. precautions System Context 주입 (1차 동일)
7. Gemini 단일 편지 생성. 3안 아님.

LLM 호출 1회.
**에러**: `400` · `403 FORBIDDEN` · `404 RESIDENT_NOT_FOUND` · `422 NO_DATA` · `502 LLM_ERROR`

### 3. POST /api/reports/send
**요청** (generate 응답을 프론트가 들고 있다가 전체 실어 보냄, 1차 send 패턴)
```json
{
  "residentId":1, "periodStart":"2026-05-18", "periodEnd":"2026-05-24",
  "recordedDays":5, "tone":"POLITE",
  "statsSummary": {}, "sourceNoticeIds":[101,105,112,118,124],
  "aiGeneratedText":"...AI 원본...",
  "finalText":"...직원 편집 최종본..."
}
```
**응답 201**: `{ "report": { "id":7, "status":"SENT", "sentAt":"..." } }`

동작: report INSERT (status=SENT, sent_at=now). `ai_generated_text`(원본)와 `final_text`(편집본) 분리 저장(1차 검증 로그 철학).
**보안**: 서버에서 `sourceNoticeIds`가 실제 해당 어르신·해당 주 것인지 재검증(클라이언트 신뢰 금지).
**에러**: `400` · `403` · `404` · `500`

### 변경 없는 API

### [S3-1] DB 마이그레이션 (report 테이블)

### AI에게 시킬 때 황금 규칙 (3차 추가)
- **report는 신규**, notice는 절대 건드리지 마. 같은 어르신을 공유할 뿐 별개 테이블.
- preview와 generate를 합치려 하면 막아라. 분리가 의도(비용·UX).
- 0일은 서버에서도 막는다. 프론트 막힘만 믿지 마.
- 통계는 코드가, 편지는 AI가. AI에게 숫자 만들게 하지 마.
- 와상 환자 박정호 테스트는 1·2·3차 모두 필수 — precautions 방어가 깨지면 즉시 알 수 있는 시금석.
