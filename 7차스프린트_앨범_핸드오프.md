# 케어알림장 — 7차 스프린트 핸드오프 (앨범) v1.0

> **이 문서의 성격**: 1~6차 + UX 미세조정 패치 + 로그인 전환 링크 패치가 **이미 구현 완료된 상태**에 얹는 **델타 문서**입니다.
>
> **7차 스프린트 범위**: 앨범 (활동 단위, 사진 묶음). 직원이 시설 활동을 만들고 참여 어르신 N명을 태깅, 사진 최대 10장 업로드. 보호자는 본인 어르신이 참여한 활동만 자동 필터링되어 조회. **AI 호출 0회** (4·6차와 같은 비-AI 스프린트). 6차 사진 인프라 재활용.
>
> **변경 총량**: 신규 테이블 2개(album, album_resident) + 신규 API 8개 + 신규 화면 4개(직원 2 + 보호자 2) + 신규 폴더 `/uploads/albums/` + 홈 카드 활성화. 1~6차 화면·API는 건드리지 않음.
>
> **⚠️ 보안 핵심**: 7차 가장 큰 보안 포인트는 **보호자가 본인 어르신 참여 활동만 보게 강제**. URL 직접 호출 시도도 서버에서 차단(`album_resident → guardian` JOIN 검증).

## 1단계. 요구사항 (확정)

### 공통
- AI 호출 0회
- 6차 사진 인프라 그대로 재활용 (폴더만 `/uploads/albums/`로 추가)
- 직원·보호자 모두 조회 가능. 작성·수정·삭제는 SOCIAL_WORKER+ADMIN (CAREGIVER 제외)

### 핵심 구조
**활동(앨범) : 사진 = 1 : N (최대 10)**, **활동 ↔ 어르신 = N : M** (`album_resident`)

| 항목 | 내용 |
|---|---|
| 활동 단위 | 키즈노트와 동일. "어버이날 행사" 같은 활동 1건 = 사진 묶음 + 참여 어르신 N명 |
| 어르신 태깅 | 활동 단위로 N명 태깅 (개별 사진별 태깅 X) |
| 보호자 노출 | 본인 어르신이 참여한 활동만 (URL 직접 호출도 차단) |
| 사진 매수 | 활동당 최대 10장 |
| 화면 형태 | 활동 카드 그리드 → 상세 페이지 사진 전체 |
| 라이트박스 | 도입 X (필요 시 추후) |

### MVP 범위 안
- 활동 등록·수정·삭제
- 활동 카드 그리드 + 월/연도 필터
- 활동 상세 페이지
- 사진 다중 업로드 (모달에서 한 번에 5~10장)
- 직원 홈·보호자 홈 앨범 카드 활성화
- 보호자 본인 어르신 참여 활동 자동 필터링

### MVP 범위 밖
- 사진 라이트박스 / 댓글 / 좋아요 / 영상 / 자동 태깅 / 검색

### 권한 정책

| 역할 | 활동 조회 | 활동 작성/수정/삭제 |
|---|---|---|
| CAREGIVER | ✓ (시설 전체) | ✗ |
| SOCIAL_WORKER | ✓ | ✓ (본인 작성 + ADMIN 예외) |
| ADMIN | ✓ | ✓ (모든 활동) |
| GUARDIAN | ✓ (본인 어르신 참여만) | ✗ |

수정·삭제는 **4차 공지사항 패턴 재사용** (본인 글 + ADMIN 예외).

## 2단계. 사용자 흐름 (요약)

```
[직원 /home]                          [보호자 /parent/home]
   └ 앨범 카드 활성                     └ 앨범 카드 활성
   ↓                                     ↓
[/albums]                             [/parent/albums]
   카드 그리드 (3열)                    본인 어르신 참여만 자동 필터
   [+ 활동 등록] (직원만)                작성·수정·삭제 UI 모두 숨김
   ↓ 카드 클릭                          ↓ 카드 클릭
[/albums/{id}]                        [/parent/albums/{id}]
   상세 + 참여 어르신 칩(호실 포함)      상세 + 참여 어르신(이름만)
   [수정][삭제] (작성자·ADMIN)
```

### 활동 등록 흐름 (모달)
```
[+ 활동 등록]
  ↓ 모달 (6차 공용 Modal 재사용)
  활동명 (필수) + 활동 날짜 (미래 허용) + 설명 (선택)
  참여 어르신 다중 체크박스 (시설 전체, 검색 input 포함)
  → 최소 1명 필수
  사진 업로드 (드래그앤드롭 + 멀티 선택)
  → 클라이언트가 1장씩 POST /api/uploads/album-photo 병렬 호출
  → 성공한 것만 누적, 실패는 토스트 안내
  → 최대 10장
  [저장]
  ↓
  POST /api/albums (활동 + 어르신 + 사진 URL 한 번에, 트랜잭션)
  ↓
  목록 갱신
```

### 보호자 권한 격리 핵심 쿼리
```sql
SELECT DISTINCT a.* FROM album a
JOIN album_resident ar ON ar.album_id = a.id
JOIN guardian g ON g.resident_id = ar.resident_id
WHERE g.user_id = :token_user_id
  AND a.facility_id = :token_facility_id
  AND a.deleted_at IS NULL
ORDER BY a.activity_date DESC
```
박현우(박정호 가족)로 로그인 시 박정호가 참여하지 않은 "실버 체조 시간"이 자동 숨김.

## 4단계. 데이터 모델 (델타)

### 신규 테이블 1: album
| 컬럼 | 타입 | 제약 |
|---|---|---|
| id | bigint | PK |
| facility_id | bigint | FK → facility |
| author_id | bigint | FK → app_user |
| activity_date | date | NOT NULL (미래 허용) |
| title | varchar(100) | NOT NULL |
| description | text | NULL (500자) |
| photos | jsonb | NOT NULL default `[]` |
| created_at/updated_at/deleted_at | timestamptz | soft delete |

**photos JSONB** (6차 meal_log 동일):
```json
[ { "url": "/static/albums/2026/05/abc.jpg", "uploadedAt": "..." } ]
```

### 신규 테이블 2: album_resident (N:M)
| 컬럼 | 타입 | 제약 |
|---|---|---|
| album_id | bigint | FK → album, ON DELETE CASCADE |
| resident_id | bigint | FK → resident |
| created_at | timestamptz | default now |

**복합 PK** `(album_id, resident_id)` — 중복 매핑 자동 차단.

### 인덱스
```sql
CREATE INDEX idx_album_facility_date
    ON album (facility_id, activity_date DESC) WHERE deleted_at IS NULL;
CREATE INDEX idx_album_author
    ON album (author_id) WHERE deleted_at IS NULL;
CREATE INDEX idx_album_resident_for_guardian
    ON album_resident (resident_id, album_id);
```
세 번째가 **보호자 권한 쿼리의 핵심**. 컬럼 순서 중요(`resident_id` 앞).

### 마이그레이션 DDL
```sql
-- 7차 스프린트: 앨범
CREATE TABLE album (
    id            BIGSERIAL PRIMARY KEY,
    facility_id   BIGINT NOT NULL REFERENCES facility(id),
    author_id     BIGINT NOT NULL REFERENCES app_user(id),
    activity_date DATE NOT NULL,
    title         VARCHAR(100) NOT NULL,
    description   TEXT,
    photos        JSONB NOT NULL DEFAULT '[]',
    created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
    deleted_at    TIMESTAMPTZ
);
CREATE INDEX idx_album_facility_date
    ON album (facility_id, activity_date DESC) WHERE deleted_at IS NULL;
CREATE INDEX idx_album_author
    ON album (author_id) WHERE deleted_at IS NULL;

CREATE TABLE album_resident (
    album_id    BIGINT NOT NULL REFERENCES album(id) ON DELETE CASCADE,
    resident_id BIGINT NOT NULL REFERENCES resident(id),
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
    PRIMARY KEY (album_id, resident_id)
);
CREATE INDEX idx_album_resident_for_guardian
    ON album_resident (resident_id, album_id);
```

### 사진 인프라 (6차 재활용)
- 저장: `{UPLOAD_DIR}/albums/{yyyy}/{mm}/{uuid}.{ext}`
- 6차 file_service 그대로 호출 (폴더 인자만 다름)
- 1장당 5MB, jpg/jpeg/png/webp, **활동당 최대 10장**
- album soft delete 시 사진 파일 물리 삭제 X

### 시드 데이터 (시연 시나리오)
- 활동 1: 어버이날 행사 (5/8) — 어르신 1·2·3·4 참여 + 사진 2장
- 활동 2: 실버 체조 시간 (5/12) — 어르신 4·5 참여 (**박정호 미참여 — 와상**)
- 활동 3: 봄나들이 산책 (5/15) — 어르신 1·2 참여
- 활동 4: 5월 생신 잔치 (5/25) — 어르신 3, **사진 0장** (사후 등록 시나리오)

**보호자 시연 검증**:
| 보호자 | 어르신 | 보이는 활동 |
|---|---|---|
| 김보람 | 김순자(1) | 어버이날·봄나들이 (2건) |
| 이지원 | 이복남(2) | 어버이날·봄나들이 (2건) |
| 박현우 | 박정호(3) | 어버이날·생신잔치 (2건). **체조 안 보임** — 와상 미참여 |

### 1~6차 원칙 일관성
- soft delete + facility_id 격리 + 부분 인덱스 + JSONB 사진 배열
- 버전 관리 없음 (4·6차와 동일)

## 5단계. API 명세

### 신규 8개 + 변경 1개
| # | 메서드 | 경로 | 의존성 |
|---|---|---|---|
| 1 | GET | /api/albums | require_staff |
| 2 | GET | /api/albums/{id} | require_staff |
| 3 | POST | /api/albums | require_content_editor |
| 4 | PATCH | /api/albums/{id} | require_content_editor + (작성자\|ADMIN) |
| 5 | DELETE | /api/albums/{id} | require_content_editor + (작성자\|ADMIN) |
| 6 | POST | /api/uploads/album-photo | require_content_editor |
| 7 | GET | /api/parent/albums | require_guardian |
| 8 | GET | /api/parent/albums/{id} | require_guardian + 권한 JOIN |
| 변경 | GET | /api/home/summary, /api/parent/me | albums 카운트 추가 |

### 1. GET /api/albums?year=&month=
```json
{
  "year":2026, "month":5, "total":4,
  "items":[
    {"id":12,"activityDate":"2026-05-08","title":"어버이날 행사",
     "description":"...","photoCount":8,"thumbnailUrl":"/static/albums/...",
     "participants":[{"id":1,"name":"김순자"},{"id":2,"name":"이복남"}],
     "author":{"id":2,"name":"박서준"},"canEdit":true,"createdAt":"..."}
  ]
}
```
- `thumbnailUrl`: photos[0]?.url 또는 null
- `canEdit`: 서버가 미리 계산
- facility_id 토큰 자동 필터

### 2. GET /api/albums/{id}
전체 album 객체. photos 전체 + participants(이름+호실) + author + canEdit.
다른 시설 활동 접근 시 403.

### 3. POST /api/albums ⭐
```json
{
  "activityDate":"2026-05-08", "title":"어버이날 행사",
  "description":"...", "residentIds":[1,2,3,4],
  "photos":[{"url":"/static/albums/2026/05/abc.jpg"}]
}
```

**서버 트랜잭션 필수**:
```python
async with db.begin():
    album = await create_album(...)
    await bulk_insert(album_resident, [(album.id, rid) for rid in residentIds])
```

**검증 (모두 필수)**:
- `residentIds.length >= 1` (EMPTY_PARTICIPANTS, 400)
- `residentIds.length <= 50`
- 각 residentId가 본인 시설 어르신인지 (INVALID_RESIDENT_FOR_FACILITY)
- `photos.length <= 10`
- 각 photo.url이 `/static/albums/` 시작 (INVALID_PHOTO_URL — 외부 URL 차단)
- title 필수, description 500자, activityDate 필수 (미래 허용)

응답 201: 생성된 album 전체.

### 4. PATCH /api/albums/{id}
권한: author_id == user_id OR role == 'ADMIN'.
residentIds 변경 시 **단순 replace** (트랜잭션 안에서 DELETE → INSERT).

### 5. DELETE /api/albums/{id}
권한 동일. soft delete. 사진·매핑 그대로 두되 조회에서 `album.deleted_at IS NULL` 필터로 숨김.

### 6. POST /api/uploads/album-photo
6차 meal-photo와 동일 패턴, 폴더만 `albums/`. **file_service 재사용**.
응답: `{ url, filename, sizeBytes }`
클라이언트는 사진 N장 선택 시 N번 병렬 호출. 각 호출 독립.

### 7. GET /api/parent/albums ⭐ (보안 핵심)
**쿼리** (idx_album_resident_for_guardian 활용):
```sql
SELECT DISTINCT a.*
FROM album a
JOIN album_resident ar ON ar.album_id = a.id
JOIN guardian g ON g.resident_id = ar.resident_id
WHERE g.user_id = :guardian_user_id
  AND g.deleted_at IS NULL
  AND a.facility_id = :facility_id
  AND a.deleted_at IS NULL
  AND a.activity_date BETWEEN :start AND :end
ORDER BY a.activity_date DESC;
```

응답에서 제외:
- `author` (작성자 직원 정보)
- `canEdit`
- `createdAt`, `updatedAt`
- 참여 어르신의 `roomNumber`

### 8. GET /api/parent/albums/{id} (권한 핵심)
**권한 체크 (필수)**:
```sql
SELECT 1 FROM album_resident ar
JOIN guardian g ON g.resident_id = ar.resident_id
WHERE ar.album_id = :album_id
  AND g.user_id = :guardian_user_id
  AND g.deleted_at IS NULL
LIMIT 1;
-- 없으면 403 FORBIDDEN
```
응답: photos 전체 + participants(이름만) + description.
author·canEdit·timestamps 응답 제외.

### AI에게 시킬 때 황금 규칙 (7차 추가)
- **6차 file_service 재사용**. 새로 만들지 마. 폴더 인자만 분기.
- **POST /api/albums는 반드시 트랜잭션**. album + album_resident 일괄.
- **인덱스 컬럼 순서 `(resident_id, album_id)`** — 반대로 만들면 보호자 쿼리 효율 ↓.
- **보호자 응답에서 author, canEdit, 다른 어르신 호실 제외**. 5차 정책 연장.
- **사진 URL은 반드시 /static/albums/ 시작 검증**. 외부 URL 차단.
- **박현우(박정호 가족) ↔ 체조 시간**은 7차 보안 시금석. URL 직접 시도해도 403.
- 라이트박스·영상·댓글은 범위 밖. 만들면 제거.
