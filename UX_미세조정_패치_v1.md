# 케어알림장 — 사용자 경험 미세 조정 패치 (UX Refinement) v1.0

> **이 문서의 성격**: 1~6차 스프린트가 **이미 구현 완료된 상태**에 얹는 **사용자 경험 다듬기 패치**입니다. 새 기능 추가는 없고, 문구·시각 표현·AI 출력 톤을 다듬는 미세 조정만 담겨 있습니다.
>
> **패치 범위**: 보호자 화면에서 AI 흔적 텍스트 제거 + 상태값 아이콘 라벨 추가 + AI 생성 본문에 자연스러운 이모지 도입. 7차 앨범으로 가기 전 단계의 "사용자 경험 완성도" 패치.
>
> **변경 총량(작음)**: AI 프롬프트 2곳 확장(알림장·리포트) + 보호자 화면 문구 다수 교체 + 상태값 표시 컴포넌트 신규 1개. 신규 테이블·신규 API·신규 화면 모두 없음.

---

## 1. 보호자 화면 — AI 흔적 텍스트 전수 제거

### 정책 (5차 정책 유지 + 강화)
보호자에게 **"AI가 작성/처리했다"는 텍스트 노출 일체 금지**. 카테고리 뱃지 같은 시각 요소는 유지 (이건 단순히 "분류 결과"로 보임).

### 교체 대상 문구 목록

| 화면 | 기존 (현재 추정) | 변경 후 | 비고 |
|---|---|---|---|
| `/parent/notices/{id}` | "(오타 교정본)" 같은 부속 표기 | **완전 제거** | 본문에 어떤 보조 표기도 없이 깔끔하게 |
| `/parent/notices/{id}` | "AI 다듬기 완료" 같은 흔적 | **제거** | 처리 과정 노출 X |
| `/parent/reports/{id}` | 동일 패턴 흔적 | **제거** | |
| `/parent/inquiries/new` | "AI가 문의 내용을 자동으로 분류하여 담당자에게 전달합니다" | **"더 나은 서비스 제공을 위해 AI를 사용 중입니다."** | 문의 작성 화면 상단 또는 [전송] 버튼 옆 안내 |
| 보호자 화면 어디든 | "AI 생성", "자동 분류", "AI 작성" 같은 표현 | **모두 제거 또는 부드러운 표현으로** | "AI"라는 단어는 위 한 곳만 |

### 교체 작업 가이드
1. 보호자 화면 컴포넌트(`pages/parent/*`)에서 정규식으로 "AI", "자동", "교정", "다듬", "분류" 같은 단어 검색
2. 각 위치마다 위 표 기준으로 판단
3. 직원 화면(`pages/Dashboard`, `pages/Reports`, `pages/Inquiries` 등)은 **건드리지 않음** — 직원에겐 AI 처리가 도구로 보여야 하므로 표현 유지

### 확인 포인트
- 보호자 알림장·리포트·문의 화면에서 페이지 전체 텍스트 검색 시 "AI"라는 단어가 위 한 곳(문의 작성 안내)에만 나옴
- "(오타 교정본)" 류 부속 표기가 어디에도 안 나옴
- 직원 화면은 1~6차 그대로 동작

## 2. 보호자 알림장·리포트 상세 — 상태값 아이콘 라벨 추가

본문 텍스트만 있어서 정보 밀도가 낮고 딱딱하다는 피드백 반영. 보호자 알림장 상세에 **건강·기분·식사·투약 상태값을 아이콘과 라벨로 시각화**해서 함께 표시.

### 5차 정책과의 관계
5차에서 "보호자 응답에서 raw_memo·structured_status 등 직원 메타 제외"로 정했었으나, **structured_status는 보호자에게도 유용한 핵심 정보**라 일부 해제. 단, raw_memo·ai_generated_texts 같은 진짜 내부 메타는 여전히 숨김.

### 백엔드 변경 (응답 확장)

`GET /api/parent/notices/{id}` 응답에 `structuredStatus` 필드 추가:

```json
{
  "notice": {
    "id": 101,
    "residentName": "김순자",
    "finalText": "...",
    "structuredStatus": {
      "health": "GOOD",
      "mood": "GOOD",
      "meal": "LITTLE",
      "medication": "DONE"
    },
    "sentAt": "...",
    "readAt": "..."
  }
}
```

`GET /api/parent/reports/{id}` 응답에 `statsSummary` 일부 노출:

```json
{
  "report": {
    "id": 7,
    "residentName": "김순자",
    "finalText": "...",
    "weekStats": {
      "recordedDays": 5,
      "topMood": "GOOD",
      "topMeal": "FULL"
    },
    "sentAt": "...",
    "readAt": "..."
  }
}
```

리포트는 한 주 요약이라 단순 카운트 노출(완식 N일 등)보다 **"이번 주 대부분 ___" 형태**가 보호자에 더 의미 있음. 그래서 `topMood`·`topMeal`은 그 주에 가장 빈도 높았던 값을 코드가 집계해 내려보냄. `recordedDays`는 "기록된 날 / 7일"로 신뢰도 표시용.

여전히 응답에 없는 것 (5차 정책 유지):
- raw_memo
- ai_generated_texts
- author 정보
- classification_scores·confidence
- 통계의 raw 숫자(메뉴별 분포 등 — 리포트만)

### 프론트 — 상태값 아이콘 라벨 컴포넌트

신규 컴포넌트 `components/StatusDisplay.tsx` (보호자 화면 전용):

| 상태 종류 | 값 | 아이콘 + 라벨 |
|---|---|---|
| **건강** | GOOD | 🌟 좋음 |
| | NORMAL | 🌤️ 보통 |
| | NEEDS_OBSERVATION | 🩺 관찰 필요 |
| **기분** | GOOD | 😊 좋음 |
| | NORMAL | 🙂 보통 |
| | ANXIOUS | 😔 다소 불편 |
| **식사** | FULL | 🍚 완식 |
| | NORMAL | 🥄 보통 |
| | LITTLE | 🌱 조금 |
| | REFUSED | — 거의 못 드심 |
| **투약** | DONE | 💊 완료 |
| | (없음) | (표시 안 함) |

색감은 보호자 화면의 따뜻한 톤(살구·미색)과 어울리게. 각 항목은 작은 카드로:

```
┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐
│ 🌟       │  │ 😊       │  │ 🌱       │  │ 💊       │
│ 건강     │  │ 기분     │  │ 식사     │  │ 투약     │
│ 좋음     │  │ 좋음     │  │ 조금     │  │ 완료     │
└─────────┘  └─────────┘  └─────────┘  └─────────┘
```

본문 텍스트(`finalText`) 바로 위 또는 아래에 배치. 가로 4칸 한 줄.

### 리포트 상세 표시 형태

리포트는 "그 주의 대표값" 표시:
```
이번 주는 대부분 🌟 좋음 / 😊 좋음 / 🍚 완식 하셨어요  ·  기록된 날 5 / 7일
```
한 줄 요약 + 본문 편지 순서. 본문이 메인이고 상태는 보조.

### 안 들어가는 값 처리
`structuredStatus.medication`이 null이거나 빈 값이면 해당 카드 표시 안 함(필수가 아님). 다른 항목은 항상 표시.

### 확인 포인트
- 보호자가 알림장 상세 진입 시 4개 카드(또는 3개) 즉시 표시
- 본문이 좀 짧아도 시각 정보가 있어 화면이 비어 보이지 않음
- 리포트 상세에서 한 줄 요약이 자연스럽게 본문 위에 놓임

## 3. AI 생성 본문에 이모지 자연스럽게

알림장·리포트 본문이 텍스트만이라 딱딱하다는 피드백. AI에게 **본문 안에 자연스러운 이모지를 적당히 섞도록** 프롬프트 확장.

### 가드레일 (중요)
이모지를 무제한 허용하면 AI가 한 문장마다 박을 수 있어 어색해짐. 다음 제약을 프롬프트에 명시:

1. **개수**: 본문 전체에 2~4개 정도. 한 문단에 1개 이하 권장.
2. **금지 카테고리**: 부정·자극적 이모지 금지 (💀 ⚰️ 🤮 🤕 등). 의료 응급·심각 의미 이모지도 금지 (🚑 🩸 등).
3. **precautions 제약 적용**: 와상 환자처럼 활동 제약이 있는 어르신에겐 활동 이모지(🚶 🏃) 사용 금지. 1차 precautions 주입 시스템에 자연히 통합.
4. **위치**: 문장 끝, 또는 문장 중간 자연스러운 곳. 문장 시작에 박지 않음.

### 안전 화이트리스트 (프롬프트 예시로 제공)

| 카테고리 | 이모지 예시 |
|---|---|
| 식사 | 🍚 🍲 🥣 🍪 ☕ |
| 활동(가능 어르신만) | 🚶 ☀️ 🌿 📖 🎵 ✏️ |
| 감정·돌봄 | 😊 ☺️ 💛 🌷 ✨ 🌟 |
| 건강·기록 | 💊 (투약 시) 🌙 (수면 시) 🩺 (관찰 필요 시) |
| 인사 | 🙏 💌 |

### 프롬프트 패치 위치

**1) `services/gemini_service.py` (1차 generate)** — 알림장 생성 프롬프트에 추가:
```
출력 작성 규칙:
- 본문에 따뜻한 이모지를 2~4개 정도 자연스럽게 배치하세요.
- 사용 가능한 이모지 예시: 🍚 ☕ ☀️ 😊 💛 🌷 💊 ✨ 🙏
- precautions에서 제약된 활동에 해당하는 이모지(🚶 🏃 등)는 절대 사용하지 마세요.
- 의료 응급·부정 이모지(💀 ⚰️ 🚑 🩸 🤮 등)는 절대 사용하지 마세요.
- 한 문단에 이모지 1개 이하, 문장 시작이 아닌 자연스러운 위치에.
```

**2) `services/report_service.py` (3차 generate_weekly_letter)** — 주간 리포트 프롬프트에 동일 규칙 추가. 리포트는 본문이 더 길어서 이모지 4~6개까지 허용해도 자연스럽지만, 동일 가드레일 적용.

### refine(다듬기) 프롬프트는 변경하지 않음
1차의 `POST /api/notices/refine`는 "맞춤법·표현 다듬기"만 함. 이모지를 추가·제거하지 않도록 기존 프롬프트 그대로 유지(아니면 직원이 직접 추가한 이모지가 사라지는 사고).

### 확인 포인트
- 박정호(와상) 어르신으로 알림장 생성 시 본문에 🚶 같은 활동 이모지가 들어가지 않음
- 일반 어르신으로 생성 시 본문에 따뜻한 이모지 2~4개가 자연스러운 위치에 들어 있음
- 본문에 💀 ⚰️ 🚑 같은 부적절 이모지가 없음
- refine(다듬기) 한 번 더 누른다고 이모지가 사라지거나 늘어나지 않음
