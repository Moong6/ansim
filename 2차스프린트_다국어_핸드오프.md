# 케어알림장 — 2차 스프린트 핸드오프 (다국어 메모 입력) v1.0

> **이 문서의 성격**: 1차 스프린트(MVP)가 **이미 구현 완료된 상태**에 얹는 **델타(변경분) 문서**입니다. 처음부터 다시 만드는 게 아니라, 기존 코드의 특정 부분만 수정/추가합니다. 1차 산출물(`개발_핸드오프_문서_v3.md`, `init_db.sql`, 작업지시문 Playbook)을 먼저 읽은 전제로 작성되었습니다.
>
> **2차 스프린트 범위**: 다국어 메모 입력 지원 **하나만**. 외국인 요양보호사가 모국어로 특이사항 메모를 입력하면 엔진이 한국어 알림장으로 변환. (스마트 문의 자동 분류는 3차로 분리)
>
> **변경 총량(작음)**: DB 컬럼 2개 + API 변경 3건 + 신규 API 1건 + S3 화면에 언어 칩 1줄. 새 테이블·새 화면·새 페이지 없음.

## 1단계. 요구사항 (확정)

| 항목 | 내용 |
|---|---|
| 기능명 | 다국어 메모 입력 지원 |
| 한 줄 정의 | 외국인 요양보호사가 모국어로 특이사항 메모를 입력해도, 엔진이 자연스러운 한국어 알림장으로 변환 |
| 타겟 | 한국어 작문이 서툰 외국인 요양보호사 (베트남·중국·필리핀 등) |
| 해결 문제 | 외국인 종사자의 한국어 작문 어려움 (1차에서 명시한 3대 문제 중 하나) |
| 핵심 동작 | 메모를 외국어로 입력(타이핑/음성) → generate가 언어 감지/적용 + 한국어로 변환하여 알림장 생성 → 결과는 항상 한국어 |
| 지원 언어 | 한국어(ko) / 베트남어(vi) / 중국어(zh) / 영어(en) — 4개로 시작 |
| 범위 안 | 자유 메모의 외국어 입력, 음성 입력 외국어 인식, 입력 언어 기록, 계정에 언어 기억 |
| 범위 밖 | 화면 UI 다국어화 / 결과물 외국어 출력 / 역번역 보기 / 보호자 번역 / 스마트 문의 |

### 핵심 전제 (왜 이렇게 작은가)
1차에서 입력을 **"정형 상태 칩 + 자유 메모"**로 분리해둔 덕분에, 칩(건강/기분/식사/투약)은 번역이 필요 없고 **자유 메모만** 외국어→한국어 변환 대상입니다. 그래서 변경이 "얹기" 수준으로 끝납니다.

## 2단계. 사용자 흐름 (1차 대비 델타)

| 지점 | 1차(v4) | 2차 |
|---|---|---|
| 메모 입력 | 한국어 textarea + 음성 | **언어 선택 칩** 추가 → 그 언어로 입력·음성 |
| AI 생성 | 메모(한국어) → 한국어 알림장 | 메모(외국어) → 한국어로 변환 → 한국어 알림장 |
| 저장 | — | 입력 언어(memo_lang) 기록 |
| 언어 기본값 | — | 로그인 직원의 preferred_lang 자동 적용 |

흐름 자체는 1차와 동일. 메모 입력 지점에 언어 칩이 하나 끼어들고, 생성 시 서버가 변환할 뿐. **결과는 항상 한국어 고정**(보호자가 한국인). **역번역 보기는 도입하지 않음**(직원이 한국어 결과를 1차와 동일하게 검수). 한국어를 선택하면 1차와 100% 동일하게 동작.

## 3단계. 화면 명세 (델타)

### 변경 1. S3 특이사항 메모 영역 — 언어 칩 추가
기존 메모 textarea **바로 위**에 입력 언어 선택 칩 추가.

| 요소 | 내용 |
|---|---|
| 라벨 | "메모 입력 언어" |
| 언어 칩 | 🇰🇷 한국어 / 🇻🇳 Tiếng Việt / 🇨🇳 中文 / 🇵🇭 English (칩 단일 선택) |
| 기본 선택 | 로그인 직원의 `preferredLang` (없으면 ko) |
| placeholder | 선택 언어에 맞춰 변경 (예: 베트남어 선택 시 베트남어 예시 문구) |
| 음성 버튼 | 선택 언어를 Web Speech API `lang`에 전달 / 해당 언어 미지원 브라우저면 버튼 숨김 |
| 칩 변경 시 | 즉시 `PATCH /api/users/me/language` 호출해 preferred_lang 저장 (별도 버튼 없음) |

상태 칩·프로그램 체크박스·S1·S2·S4(3안 선택/전송/4대 방어)는 **전부 1차 그대로**. 변경 없음.

### 변경 2. 로그인 직후 언어 기본값 적용
login 응답의 `user.preferredLang`을 S3 언어 칩 기본 선택값으로 사용.

## 4단계. 데이터 모델 (델타)

### app_user 테이블 — 컬럼 추가
| 컬럼 | 타입 | 제약 | 설명 |
|---|---|---|---|
| preferred_lang | varchar(10) | NOT NULL default 'ko' | 직원 기본 메모 입력 언어. 칩 변경 시 즉시 저장 |

### notice 테이블 — 컬럼 추가
| 컬럼 | 타입 | 제약 | 설명 |
|---|---|---|---|
| memo_lang | varchar(10) | NOT NULL default 'ko' | 이 알림장 메모의 입력 언어. 발송 시 기록 |

언어 코드는 ISO 639-1: `ko` / `vi` / `zh` / `en`. enum이 아닌 varchar(확장 유연성).

### 검증 로그 관점
`raw_memo`(외국어 원문 그대로 보존) + `memo_lang`(언어) + `final_polished_text`(한국어 결과)가 한 행에 남아, "외국어 원문 → 한국어 결과" 변환 품질 검증 로그가 자연히 쌓임. 1차의 "데이터 4단 분리" 철학 연장.

### 마이그레이션 DDL
```sql
-- 2차 스프린트: 다국어 입력 지원 (기존 DB에 적용)
ALTER TABLE app_user ADD COLUMN preferred_lang VARCHAR(10) NOT NULL DEFAULT 'ko';
ALTER TABLE notice   ADD COLUMN memo_lang      VARCHAR(10) NOT NULL DEFAULT 'ko';
```
> 운영 중 DB라면 위 ALTER만 적용(init_db.sql 재실행 금지 — 데이터 소실). 정석은 Alembic 마이그레이션 파일로 관리.

### 시드 보강 (데모용, 선택)
외국인 직원 1명 추가 권장:
```sql
INSERT INTO app_user (facility_id, email, password_hash, name, role, preferred_lang)
VALUES (1, 'huong@happy.kr', '$2b$12$REPLACE_WITH_BCRYPT_HASH', '후엉', 'CAREGIVER', 'vi');
-- 이후 assignment로 어르신 몇 명 담당 배정
```

## 5단계. API 명세 (델타)

### 변경·추가 요약
| 구분 | 엔드포인트 | 변경 |
|---|---|---|
| 변경 | POST /api/auth/login | 응답 user에 `preferredLang` 추가 |
| 신규 | PATCH /api/users/me/language | preferred_lang 즉시 저장 |
| 변경 | POST /api/notices/generate | 요청에 `memoLang` 추가 + 언어 변환 로직 |
| 변경 | POST /api/notices/send | 요청에 `memoLang` 추가 (notice 저장) |

### 1. POST /api/auth/login (변경)
응답 `user`에 `preferredLang` 추가:
```json
{ "token":"...", "user": { "id":4,"name":"후엉","role":"CAREGIVER",
  "preferredLang":"vi", "facility":{"id":1,"name":"행복요양원"} } }
```

### 2. PATCH /api/users/me/language (신규)
요청 `{ "preferredLang": "vi" }` → 응답 200 `{ "preferredLang": "vi" }`
허용 값 `ko|vi|zh|en`. `/me`이므로 토큰 본인 계정만 변경 가능.
에러: `400 INVALID_LANGUAGE` · `401 UNAUTHORIZED` · `500`

### 3. POST /api/notices/generate (변경) ⭐
요청에 `memoLang` 추가:
```json
{ "residentId":1, "participatedProgramIds":[1,3],
  "status":{"health":"GOOD","mood":"GOOD","meal":"LITTLE","medication":"DONE"},
  "memo":"Ăn hết nửa bát canh, tâm trạng vui vẻ",
  "memoLang":"vi", "tone":"POLITE" }
```
응답(1차 구조 + `detectedLang`):
```json
{ "drafts":[{"index":0,"label":"A","text":"안녕하세요 보호자님..."}, ...],
  "softenedCount":0, "appliedPrograms":[...], "detectedLang":"vi" }
```
**서버 로직 (1차 generate에 언어 처리만 얹음)**:
1. `memoLang` 있으면 그 언어로 간주, 없으면(null) 자동 감지 → `detectedLang`으로 반환
2. 메모가 외국어면 의미 파악해 한국어로 변환 (별도 번역 API 없이 Gemini 프롬프트 내 처리)
3. 이후 1차 동일: `precautions` System Context 주입(할루시네이션 방어) → 상태/프로그램/톤 결합 → Structured Output 한국어 3안
4. **출력은 반드시 한국어로 강제** (프롬프트 명시)
- LLM 호출 횟수 1차와 동일(생성 1회).
- 에러: 1차 동일 + `400 UNSUPPORTED_LANGUAGE`(허용 목록 밖)

### 4. POST /api/notices/send (변경)
요청에 `memoLang` 추가 → notice.memo_lang 저장. 나머지(append-only 버전관리 등) 1차 동일.

### 변경 없는 API
GET /api/programs/today, GET /api/residents/assigned, GET /api/residents/{id}, GET /api/notices/{id}, POST /api/notices/refine — **전부 1차 그대로**. (refine은 입력이 이미 한국어라 언어 처리 불필요)
