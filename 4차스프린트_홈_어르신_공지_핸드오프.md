# 케어알림장 — 4차 스프린트 핸드오프 (홈 + 어르신 + 공지사항) v1.0

> **이 문서의 성격**: 1차(MVP)·2차(다국어)·3차(주간 리포트)가 **이미 구현 완료된 상태**에 얹는 **델타 문서**입니다. 1~3차 산출물을 먼저 읽은 전제로 작성되었습니다.
>
> **4차 스프린트 범위**: 운영 도구 3종 — ① 홈 그리드(메인 진입점 재정의) ② 어르신 특이사항 관리 ③ 공지사항 게시판. **AI 호출 없음.** (보호자 문의 AI 분류는 5차, 식단·일정은 6차, 앨범은 7차)
>
> **변경 총량**: 신규 테이블 1개(`notice_board`) + 신규 API 7개 + 신규 화면 3개 + 라우팅 변경(로그인 후 첫 진입은 `/home`). 기존 1·2·3차 API와 화면은 **건드리지 않음**(단, 헤더에 [홈] 링크 추가 정도는 가능).

---

## 1단계. 요구사항 (확정)

### 공통 전제
- 직원용 데스크탑/태블릿 (1·2·3차와 동일)
- 1·2·3차 자산 그대로 재활용 (인증, JWT, CORS, Pydantic, SQLAlchemy, Tailwind+Vite)
- 4차에서 AI 호출은 **없음** (운영 도구 스프린트)

### 기능 1. 홈 그리드 (메인 진입점 재정의)
| 항목 | 내용 |
|---|---|
| 한 줄 정의 | 로그인 후 첫 진입 화면. 8개 기능 카드 그리드 |
| 라우팅 | 로그인 후 → `/home` (신규). 기존 `/dashboard`(알림장)는 그대로, 카드 하나의 목적지 |
| 카드 8개 | 알림장(1차) · 주간리포트(3차) · 어르신(4차) · 공지사항(4차) · **앨범·일정표·식단표·보호자문의(비활성)** |
| 카드 상태 | 활성 4개는 카운트 요약 + 미확인 점. 비활성 4개 클릭 시 `toast("준비 중입니다")` |

### 기능 2. 어르신 (특이사항 관리)
| 항목 | 내용 |
|---|---|
| 한 줄 정의 | 담당 어르신의 `precautions` 보기·편집. AI 할루시네이션 방어 데이터를 직원이 직접 제어 |
| 화면 | 좌·우 2단 (좌: 담당 목록 / 우: 선택 어르신 편집) |
| 편집 범위 | **`precautions`만** (이름·등급·호실 등은 관리자 영역, MVP 범위 밖) |
| 신규 테이블 | 없음 (1차 resident 그대로) |
| 신규 API | `PATCH /api/residents/{id}` 1개 (precautions만, 화이트리스트) |
| 핵심 가치 | 발표 시 "지금 주의사항을 추가하면 다음 알림장 생성부터 AI가 반영"을 라이브 시연 가능 |

### 기능 3. 공지사항 (게시판)
| 항목 | 내용 |
|---|---|
| 한 줄 정의 | 시설 단위 공지를 작성·조회. AI 사용 안 함 |
| 카테고리 | **"일반" 단일** (중요·핀 고정·필터 모두 X) |
| 권한 | 본인 글만 본인이 수정·삭제. **ADMIN은 모든 글 가능** (시드 ADMIN 계정 활용) |
| 신규 테이블 | `notice_board` (1차 `notice`와 이름 충돌 회피) |
| 신규 API | `GET /api/board`, `GET /api/board/{id}`, `POST /api/board`, `PATCH /api/board/{id}`, `DELETE /api/board/{id}` — 5개 |
| 범위 밖 | 댓글, 첨부파일, 푸시 알림, 검색, 보호자 화면 |

## 2단계. 사용자 흐름

```
[로그인]
    │
    ▼
[/home] 홈 그리드 (활성 4 / 비활성 4)
    │
    ├ 알림장 카드        → /dashboard (1·2차)
    ├ 주간 리포트 카드    → /reports (3차)
    ├ 어르신 카드        → /residents (4차)
    ├ 공지사항 카드      → /board (4차)
    └ 비활성 4개         → toast("준비 중입니다") + 홈 유지

[/residents] 좌: 담당 어르신 목록  /  우: 선택된 어르신
    └ precautions 편집 → 저장 시 토스트
       "수정되었습니다. 다음 알림장 생성부터 반영됩니다"

[/board]
    ├ 목록(최신순) + [새 공지 작성]
    ├ /board/{id} 상세
    └ /board/new, /board/{id}/edit (별도 페이지)

[방어 로직]
    ├ 어르신 화면: precautions 수정 후 다른 어르신 클릭 시 confirm
    │            "저장하지 않은 변경사항이 있습니다. 이동하시겠습니까?"
    └ 공지 삭제: 인라인 confirm → soft delete (deleted_at)
```

### 화면 3. /board (공지사항)

| 화면 | 요소 | 비고 |
|---|---|---|
| 목록 `/board` | 카드 리스트(최신순, 본문 첫 줄 1줄로 잘라 표시) + [새 공지 작성] | 본인 글에만 [수정][삭제] 아이콘 |
| 상세 `/board/{id}` | 제목·작성자·작성일·본문 + [목록으로] | 본인 글일 때 [수정][삭제] |

## 4단계. 데이터 모델

### 변경 없음: resident
1차 컬럼(name, room_number, care_level, precautions, birth_date, gender) 그대로. 컬럼 추가 안 함.

### 신규 테이블: notice_board

| 컬럼 | 타입 | 제약 | 설명 |
|---|---|---|---|
| id | bigint | PK | |
| facility_id | bigint | FK → facility | 시설 단위 격리 |
| author_id | bigint | FK → app_user | 작성자 |
| title | varchar(200) | NOT NULL | 제목 |
| content | text | NOT NULL | 본문 |
| created_at | timestamptz | NOT NULL default now | |
| updated_at | timestamptz | NOT NULL default now | |
| deleted_at | timestamptz | NULL | soft delete (1차 패턴) |

테이블명 `notice_board`로 명명한 이유: 1차 `notice`(일간 알림장)와 충돌 회피. SQLAlchemy 모델명은 `NoticeBoard`.

### 인덱스
| 인덱스 | 목적 |
|---|---|
| `(facility_id, created_at DESC) WHERE deleted_at IS NULL` | 목록 조회 가속(최신순) + 소프트 삭제 제외 |

### 1차·3차 원칙 일관성
- soft delete(`deleted_at`) 유지
- `facility_id`로 시설 단위 격리 (다중 시설 확장 대비)
- 부분 인덱스로 활성 데이터만 인덱싱
- **버전 관리는 적용 안 함** — 공지는 1차 notice/3차 report와 달리 "AI 산출물 vs 사람 산출물" 분리가 무의미. 단순 UPDATE.

### 마이그레이션 DDL
```sql
-- 4차 스프린트: 공지사항 게시판
CREATE TABLE notice_board (
    id           BIGSERIAL PRIMARY KEY,
    facility_id  BIGINT NOT NULL REFERENCES facility(id),
    author_id    BIGINT NOT NULL REFERENCES app_user(id),
    title        VARCHAR(200) NOT NULL,
    content      TEXT NOT NULL,
    created_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
    deleted_at   TIMESTAMPTZ
);
CREATE INDEX idx_notice_board_facility_recent
    ON notice_board (facility_id, created_at DESC)
    WHERE deleted_at IS NULL;
```
운영 중 DB에 `CREATE TABLE`만 추가. 기존 테이블 영향 없음.

### 시드 (데모용)
작성자 분산해 권한 시연 가능하게:
```sql

(1, 3, '감염병 예방 수칙 재안내',     '환절기를 맞아 감염병 예방 수칙을 다시 한 번 안내드립니다...'),
(1, 2, '정기 회의 일정 변경',         '이번 주 정기 회의가 수요일 오후 3시로 변경되었습니다.'),
(1, 3, '신규 식자재 공급업체 변경',   '4월부터 식자재 공급업체가 변경됩니다. 식단 운영에 참고 바랍니다.'),
```

## 5단계. API 명세

### 신규 7개 요약
| # | 메서드 | 경로 | 역할 | LLM |
|---|---|---|---|---|
| 1 | GET | /api/home/summary | 홈 4카드 카운트 일괄 | X |
| 2 | PATCH | /api/residents/{id} | precautions 편집(화이트리스트) | X |
| 3 | GET | /api/board | 공지 목록 | X |
| 4 | GET | /api/board/{id} | 공지 단건 | X |
| 5 | POST | /api/board | 공지 작성 | X |
| 6 | PATCH | /api/board/{id} | 공지 수정 | X |
| 7 | DELETE | /api/board/{id} | 공지 소프트 삭제 | X |

### 1. GET /api/home/summary
응답 200:
```json
{
  "alimjang":  { "todayTotal": 8, "todayCompleted": 3 },
  "report":    { "lastWeekAvailable": true },
  "board":     { "unreadCount": 2 },
  "residents": { "assignedCount": 8 }
}
```
비활성 카드 4개는 응답에 안 담음. 프론트는 그 카드들엔 토스트만.
에러: `401` · `500`

### 2. PATCH /api/residents/{id}
요청 (precautions만 허용 — Pydantic으로 화이트리스트):
```json
{ "precautions": "와상 환자, 휠체어 사용 필수..." }
```
**서버에서 다른 필드(name·birth_date 등) 시도 시 400 INVALID_FIELD로 거부.** Pydantic 모델 `ResidentUpdateSchema { precautions: str }`만 정의하면 자동.

응답 200: 수정된 resident 객체(updatedAt 갱신).
에러: `400 INVALID_FIELD` · `401` · `403 FORBIDDEN`(담당 아님) · `404 RESIDENT_NOT_FOUND`

### 3. GET /api/board (목록)
쿼리: `?limit=20&offset=0`
응답 200:
```json
{
  "total": 12,
  "items": [
    { "id": 5, "title": "...", "preview": "본문 첫 100자...",
      "author": { "id": 2, "name": "박서준", "role": "SOCIAL_WORKER" },
      "createdAt": "...", "canEdit": true }
  ]
}
```
- `preview`: 서버에서 본문 첫 100자 잘라 내림(트래픽 절약)
- `canEdit`: 서버가 토큰 user_id == author_id OR role == 'ADMIN'으로 계산해 미리 내려줌
- facility_id 토큰 자동 필터, soft delete 제외
에러: `401`

### 4. GET /api/board/{id}
응답 200: post 전체(content 포함, canEdit 포함). 다른 시설 글 → 403. 삭제된 글 → 404.

### 5. POST /api/board
요청 `{ "title", "content" }` (facility_id·author_id는 서버가 토큰에서 자동 주입)
응답 201: 생성된 post.
에러: `400 VALIDATION_ERROR`(빈 값) · `401`

### 6. PATCH /api/board/{id}
요청: title·content 중 변경할 것만.
**권한 검증 (서버 의존성으로 공통화)**:
```python
if post.author_id == current_user.id: 허용
elif current_user.role == 'ADMIN': 허용
else: 403 FORBIDDEN
```
응답 200: 수정된 post(updatedAt 갱신).
에러: `400` · `401` · `403 FORBIDDEN` · `404 POST_NOT_FOUND`

### 7. DELETE /api/board/{id}
권한 검증은 PATCH와 동일. soft delete (`deleted_at = now()`).
응답 204 No Content.
에러: `401` · `403` · `404`

### 변경 없는 API
1·2·3차의 모든 엔드포인트 그대로.

```
- 수정: 라우터 진입점(로그인 후 /home으로), 헤더에 [홈] 링크 추가
```

### AI에게 시킬 때 황금 규칙 (4차 추가)
- 4차는 **AI 호출 0회 스프린트**. Gemini 부르는 코드 추가하면 안 됨.
- `notice_board`와 `notice`는 다른 테이블. 헷갈리면 1차 코드가 무너진다.
- `PATCH /api/residents/{id}`는 **precautions 전용**. 다른 필드 허용하면 보안 사고.
- 권한 검증은 항상 서버에서. 프론트의 canEdit은 UI 표시용일 뿐.
- 비활성 카드를 진짜 기능으로 만들지 마. 토스트가 끝.
- 박정호(와상) 어르신 precautions 편집 → 알림장 반영 확인은 4차의 핵심 시연. 발표 데모도 이걸로.
