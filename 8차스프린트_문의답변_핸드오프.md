# 케어알림장 — 8차 스프린트 핸드오프 (문의 답변 작성) v1.0

> **이 문서의 성격**: 1~7차 + UX 패치 + 로그인 전환 패치가 **이미 구현 완료된 상태**에 얹는 **델타 문서**입니다.
>
> **8차 스프린트 범위**: 5차에서 미뤘던 보호자 문의 답변 작성 기능. 직원 전체(CAREGIVER+SOCIAL_WORKER+ADMIN)가 답변하고, 보호자가 본인 문의의 답변을 확인. 1 문의 = 1 답변. **AI 호출 0회** (AI 답변 초안은 9차로 분리).
>
> **변경 총량**: 신규 테이블 1개(inquiry_answer) + enum 값 추가(ANSWERED) + inquiry 컬럼 추가(answer_read_at) + 신규 API 3개 + 기존 API 4건 응답 확장 + 신규 화면 0개(5차 화면에 답변 영역 추가). 1~7차 기존 화면·API는 영향 최소.
>
> **⚠️ 정합성 핵심**: 답변 등록·삭제 시 `inquiry.status` 자동 전이 — 반드시 트랜잭션 안에서 일괄 처리. 부분 실패 시 "답변 있는데 상태는 READ" 같은 모순 상태 방지.

## 1단계. 요구사항 (확정)

### 스프린트 한 줄 정의
보호자 문의에 직원이 답변을 작성하고, 보호자가 본인 문의의 답변을 확인할 수 있게 하여 케어 커뮤니케이션의 양방향 사이클을 완성한다.

### 두 축의 작업

**축 1. 직원 답변 작성**
| 항목 | 내용 |
|---|---|
| 권한 | 직원 전체 (CAREGIVER + SOCIAL_WORKER + ADMIN) — 5차 require_staff 그대로 |
| 답변 개수 | 1 문의 = 1 답변 (스레드 X) |
| 수정·삭제 | 답변자 본인 + ADMIN만 |
| 작성 UI | 5차 문의 상세(`/inquiries/{id}`)에 답변 영역 추가 |

**축 2. 상태 흐름 확장**
| 상태 | 의미 | 전이 |
|---|---|---|
| UNREAD (5차) | 보호자 작성, 직원 미확인 | → 직원이 [확인 완료] 클릭 시 READ |
| READ (5차) | 직원 확인했으나 답변 미작성 | → 직원이 답변 등록 시 ANSWERED |
| **ANSWERED (신규)** | 답변 완료 | (답변 삭제 시 READ로 복귀) |

### 보호자 화면 변화
- 문의 상세에 답변 영역 추가
- 답변 미확인 = "답변 도착" + 빨간 점
- 보호자 상세 진입 시 `answer_read_at` 자동 갱신 (1차 `notice.read_at` 패턴)

### MVP 범위 안
- 직원이 문의 상세에서 답변 작성
- 답변 등록 시 inquiry.status 자동 ANSWERED
- 보호자 문의 상세에 답변 표시
- 답변 수정·삭제 (본인 + ADMIN)
- 직원 목록 상태 필터에 [답변 대기만][답변 완료] 추가
- 보호자 홈에 "답변 도착 N건" 카운트

### MVP 범위 밖
- **AI 답변 초안 자동 생성** (9차로 분리)
- 답변에 사진·파일 첨부
- 보호자 재문의(스레드)
- 답변 알림(이메일·푸시·SMS)
- 답변 검색·통계

### 5차 정책과의 관계
- 5차 보호자 응답에서 "author 제외" 정책 그대로 유지. **답변에 한해 author 노출은 의도된 예외**(1:1 소통 신뢰).
- 5차 `confidence`·`classification_status` 미노출 정책 그대로.
- AI 메타데이터 없음(답변은 사람 작성).
- 5차 `PATCH /api/inquiries/{id}/read`(확인 처리)는 그대로 유지. 답변은 그 다음 단계.

### 권한 정책

| 역할 | 답변 작성 | 답변 수정·삭제 |
|---|---|---|
| CAREGIVER | ✓ | 본인 답변만 |
| SOCIAL_WORKER | ✓ | 본인 답변만 |
| ADMIN | ✓ | 모든 답변 |
| GUARDIAN | ✗ | ✗ |

## 2단계. 사용자 흐름 (요약)

### 직원 워크플로우 (3단계)
```
UNREAD 상태 진입
  ├ 문의 본문 + [확인 완료로 표시] 버튼
  ↓ PATCH /api/inquiries/{id}/read
READ 상태
  ├ 답변 작성 폼 자동 노출
  ├ [답변 등록] 버튼
  ↓ POST /api/inquiries/{id}/answer
ANSWERED 상태
  ├ 답변 본문 표시 (연한 초록 배경)
  ├ 답변자 카드 (아바타·이름·답변일)
  └ 본인/ADMIN만 [✏ 수정][🗑 삭제] 노출
```

### 답변 폼 노출 정책
**READ 상태에서만 답변 폼 노출**. UNREAD에서 답변 시도 시 서버가 400 INVALID_INQUIRY_STATUS로 거부. "먼저 [확인 완료]로 표시한 뒤 답변하세요" 안내.

### 답변 수정 — inline edit
모달 X. 답변 표시 영역에 [✏ 수정] 클릭 시 같은 자리가 textarea + [저장][취소]로 변환. 4·7차 모달 패턴과 의도적으로 다름 — 문의 본문이 가려지면 답변 맥락이 흐려짐.

### 답변 삭제 후 상태
ANSWERED → **READ로 복귀** (UNREAD 아님). 확인은 했고 답변만 취소된 상태가 정확.

### 보호자 흐름
```
/parent/inquiries
  ├ 목록: hasNewAnswer 우선 → 최신순
  ├ 카드: "답변 도착" 뱃지 + 답변 첫 줄 미리보기 + 빨간 점
  ↓
/parent/inquiries/{id}
  ├ 본인 문의 본문 (5차 그대로)
  ├ 답변 영역:
  │   없으면 "직원이 곧 답변드릴 예정입니다"
  │   있으면 본문 + 직원명·직책 (예: "김민지 요양보호사") + 답변일
  ├ 답변 아래 "추가 문의는 새 문의 작성" 안내
  └ 상세 진입 시 answer_read_at 자동 갱신
```

### LLM 호출 0회
8차도 비-AI 스프린트 (4·6·7차와 동일).

## 4단계. 데이터 모델 (델타)

### 결정: 답변을 별도 테이블로 분리
inquiry에 answer_text 컬럼 추가 안 함. 답변 메타데이터(작성자·시각) + soft delete + 9차 AI 초안 컬럼 확장 위해 분리.

### enum 확장
```sql
ALTER TYPE inquiry_status ADD VALUE 'ANSWERED';
```

PostgreSQL은 `ALTER TYPE ADD VALUE` 지원. 5차의 GUARDIAN 추가 패턴 동일. **마이그레이션과 코드 배포 순서 주의** — enum 추가 트랜잭션 안에서 새 값 INSERT 동시 시도 안 됨, 마이그레이션 commit 후 코드 배포.

### inquiry 컬럼 추가
| 컬럼 | 타입 | 제약 | 설명 |
|---|---|---|---|
| answer_read_at | timestamptz | NULL | 보호자가 답변을 읽은 시각. 1차 notice.read_at 패턴 |

기존 5차 `read_at`은 **직원 확인 시각**으로 의미 다름. 두 컬럼 별개.

```sql
ALTER TABLE inquiry ADD COLUMN answer_read_at TIMESTAMPTZ;
```

### 신규 테이블: inquiry_answer

| 컬럼 | 타입 | 제약 |
|---|---|---|
| id | bigint | PK |
| inquiry_id | bigint | FK → inquiry, NOT NULL |
| author_id | bigint | FK → app_user, NOT NULL |
| content | text | NOT NULL (1~500자) |
| created_at/updated_at/deleted_at | timestamptz | soft delete |

**UNIQUE 부분 인덱스**: 활성 답변은 문의당 1건이지만 삭제 이력은 보존.

```sql
CREATE TABLE inquiry_answer (
    id          BIGSERIAL PRIMARY KEY,
    inquiry_id  BIGINT NOT NULL REFERENCES inquiry(id),
    author_id   BIGINT NOT NULL REFERENCES app_user(id),
    content     TEXT NOT NULL,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
    deleted_at  TIMESTAMPTZ
);
CREATE UNIQUE INDEX uq_inquiry_answer_active
    ON inquiry_answer (inquiry_id) WHERE deleted_at IS NULL;
CREATE INDEX idx_inquiry_answer_author
    ON inquiry_answer (author_id) WHERE deleted_at IS NULL;
```

### 상태 전이 정합성

**어플리케이션 레벨**에서 트랜잭션으로 동시 처리. DB 트리거 X.

| 동작 | inquiry_answer | inquiry.status |
|---|---|---|
| 답변 등록 | INSERT | UNREAD/READ → ANSWERED |
| 답변 수정 | UPDATE | 변경 없음 |
| 답변 삭제 | UPDATE deleted_at | ANSWERED → READ (UNREAD 아님) |

### 마이그레이션 DDL 전체

```sql
-- 8차 스프린트: 문의 답변

-- 1) enum 확장
ALTER TYPE inquiry_status ADD VALUE 'ANSWERED';

-- 2) inquiry 컬럼 추가
ALTER TABLE inquiry ADD COLUMN answer_read_at TIMESTAMPTZ;

-- 3) inquiry_answer 테이블
CREATE TABLE inquiry_answer (
    id          BIGSERIAL PRIMARY KEY,
    inquiry_id  BIGINT NOT NULL REFERENCES inquiry(id),
    author_id   BIGINT NOT NULL REFERENCES app_user(id),
    content     TEXT NOT NULL,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
    deleted_at  TIMESTAMPTZ
);
CREATE UNIQUE INDEX uq_inquiry_answer_active
    ON inquiry_answer (inquiry_id) WHERE deleted_at IS NULL;
CREATE INDEX idx_inquiry_answer_author
    ON inquiry_answer (author_id) WHERE deleted_at IS NULL;
```

### 시드 보강

5차 시드 3건 중 박현우(VISIT) 문의에 답변 시드 추가:

```sql
INSERT INTO inquiry_answer (inquiry_id, author_id, content)
VALUES (
  (SELECT id FROM inquiry WHERE content LIKE '이번 주 토요일%'),
  2,  -- 박서준 사회복지사
  '안녕하세요 보호자님. 토요일 오후 2시 방문 가능하십니다. 면회실에서 30분 정도 시간 마련해 두겠습니다. 따뜻한 옷차림으로 오시면 정원에서도 함께 시간 보내실 수 있습니다.'
);

UPDATE inquiry SET status = 'ANSWERED'
WHERE content LIKE '이번 주 토요일%';
```

시드 3건의 상태 분포:

| 보호자 | 카테고리 | 8차 상태 | 시연 가치 |
|---|---|---|---|
| 김보람 | HEALTH | UNREAD | 직원 확인 → 답변 작성 흐름 시연 |
| 이지원 | OTHER | UNREAD | 보조 케이스 |
| 박현우 | VISIT | **ANSWERED** | 보호자 답변 확인 시연 |

발표 시점에 세 상태가 모두 시드에 들어 있어 한 번에 시연 가능.

### 9차 확장 단서
inquiry_answer에 향후 추가될 컬럼 자리:
- `ai_draft text NULL` (AI 생성 초안 원본)
- `ai_used boolean NOT NULL default false` (AI 초안 채택 여부)
- `refinement_count int NOT NULL default 0` (다듬기 사용 횟수)

8차에선 만들지 않음. 9차에서 ALTER로 추가.

### 1. POST /api/inquiries/{id}/answer ⭐

**요청**: `{ "content": "..." }`

**서버 검증 (모두 필수)**:
- inquiry가 본인 시설인지 (403)
- inquiry.status가 **READ**인지 — UNREAD/ANSWERED면 400 INVALID_INQUIRY_STATUS
- content 1~500자
- 활성 답변(`deleted_at IS NULL`) 존재 시 409 ANSWER_ALREADY_EXISTS

**서버 로직 — 트랜잭션 필수**:

### AI에게 시킬 때 황금 규칙 (8차 추가)
- **답변 등록·삭제는 반드시 트랜잭션**. answer + inquiry.status 일괄 처리.
- **답변 등록은 READ 상태에서만**. UNREAD에 시도 시 400 INVALID_INQUIRY_STATUS.
- **답변 삭제 시 ANSWERED → READ** (UNREAD 아님). 직원 확인 사실은 유지.
- **UNIQUE 부분 인덱스로 1 문의 1 답변** 보장. 활성 답변만 유일성 검사.
- **5차 응답에 필드만 추가**. 기존 필드 절대 제거하지 마.
- **author.role을 답변에 포함**. 보호자에게 "김민지 요양보호사"처럼 직책 노출.
- **author.id는 보호자 응답에 제외** (5차 정책). name·role만.
- **answer_read_at은 보호자 진입 시 자동 갱신**, read_at(직원 확인용)은 그대로.
- **답변 수정은 inline edit**, 모달 X (4·7차와 다른 패턴, 의도적).
- **enum ALTER 후 코드 배포 순서** — 마이그레이션 commit 후 새 enum 값 INSERT 가능.
- 박현우(박정호 가족) ↔ 면회 답변은 8차 시연 시금석. 시드에 미리 ANSWERED 박혀 있음.
- AI 답변 초안은 9차로 분리. 8차에서 만들면 잘못.
