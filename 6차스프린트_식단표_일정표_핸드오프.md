# 케어알림장 — 6차 스프린트 핸드오프 (식단표 + 일정표) v1.0

> **이 문서의 성격**: 1차(MVP) · 2차(다국어) · 3차(주간 리포트) · 4차(홈+어르신+공지) · 5차(보호자 채널+AI 분류)가 **이미 구현 완료된 상태**에 얹는 **델타 문서**입니다. 1~5차 산출물을 먼저 읽은 전제로 작성되었습니다.
>
> **6차 스프린트 범위**: 두 운영 콘텐츠 — ① 식단표(meal_log, 하루 단위, 사진+텍스트) ② 일정표(schedule_event, 월별 목록, 공휴일·시설행사·생일). **AI 호출 없음**(4차와 같은 비-AI 스프린트). 사진 업로드 인프라가 처음 등장.
>
> **변경 총량**: 신규 테이블 2개(meal_log, schedule_event) + enum 2종 + 신규 API 11개 + 신규 화면 2개(직원) + 2개(보호자) + 파일 업로드 인프라(로컬 `/uploads`) + 신규 권한 의존성 `require_content_editor`. 기존 1~5차 화면·API는 건드리지 않음(단, /api/home/summary와 /api/parent/me에 카운트 2개 추가).

## 1단계. 요구사항 (확정)

### 공통
- AI 호출 0회 (운영 콘텐츠 스프린트)
- 직원·보호자 모두 조회 가능. 작성·수정·삭제는 SOCIAL_WORKER + ADMIN만(CAREGIVER 제외)
- 5차 require_guardian / require_staff + 신규 `require_content_editor`(SOCIAL_WORKER+ADMIN) 의존성 추가

### 기능 1. 식단표 (meal_log)
| 항목 | 내용 |
|---|---|
| 성격 | "오늘 드신 것" 사후 기록 (계획표 아님) |
| 데이터 단위 | 날짜 + 식사구분 = 1행. 한 날 최대 4건(아침/점심/저녁/간식) |
| 핵심 요소 | 사진 1~2장 + 메뉴 텍스트(줄단위) |
| 표시 형태 | **하루 단위**. 좌우 화살표로 일자 이동. 4행 세로 쌓기(아침→점심→저녁→간식) |
| 미래 날짜 | **이동·등록 차단**(서버에서도 검증) |
| 사진 클릭 | **라이트박스 X**, 그 자리에 정적 이미지로만 |
| 사진 저장 | 로컬 파일 시스템 `/uploads` (운영 시 S3 교체 단서) |
| 권한 | 작성·수정·삭제 = SOCIAL_WORKER + ADMIN |

### 기능 2. 일정표 (schedule_event)
| 항목 | 내용 |
|---|---|
| 표시 형태 | 월별 목록 (캘린더 그리드 X). 캡처 그대로 |
| 유형 | FACILITY_EVENT(시설 행사) / BIRTHDAY(어르신 생일) / HOLIDAY(공휴일) |
| 공휴일 | 시드에 2026년 한국 공휴일 14건 직접 입력. 외부 API 없음 |
| 생일 | 직원 수동 입력 + 시드 3건. 어르신 자동 추출 X |
| 시각 구분 | 공휴일=빨강🚩 / 생일=핑크🎁 / 시설행사=파랑🎉 |
| 권한 | 작성·수정·삭제 = SOCIAL_WORKER + ADMIN |

### MVP 범위 안
- 식단·일정 등록·조회·수정·삭제
- 사진 업로드 (1장당 5MB, 최대 2장, jpg/jpeg/png/webp)
- 보호자 읽기 전용 노출
- 홈 카드 활성화 + 카운트

### MVP 범위 밖
- 사진 라이트박스/갤러리 (7차)
- 캘린더 달력 그리드 UI
- 외부 공휴일 API
- 어르신 생일 자동 추출
- 시간 단위 일정 / 반복 일정 / 알림
- 식단 영양·알레르기 정보
- 사진 다수 업로드(2장 제한)
- 식단 미래 날짜 등록

```
   ↓                                     ↓
```

### 식단표 흐름 (하루 단위)
```
/meals
   └ 날짜 네비 [◀] 2026.5.26 화 [▶] [오늘로]
        · 미래 날짜 이동 차단 (오늘 이후 [▶] 비활성)
        · 키보드 ←/→ 지원
   └ 4행 세로:
        🌅 아침 / 🍚 점심 / 🌙 저녁 / 🍪 간식
        · 채워진 행: 사진(좌, 200px) + 라벨 뱃지 + 메뉴 텍스트(우) + 수정·삭제 아이콘(직원)
        · 빈 행 (직원): "🌅 아침 식단 등록하기 [+]"
        · 빈 행 (보호자): "🌅 아침 등록 예정" 회색 텍스트
   └ [+ 식단 등록] (직원만): 등록 모달
   └ 행 클릭 / 등록 모달:
        · 날짜 (datepicker, max=오늘)
        · 식사구분 (라디오)
        · 사진 1~2장 업로드 (각 5MB 이내)
        · 메뉴 텍스트 (줄단위)
        · [취소] [저장] / [삭제]
```

### 일정표 흐름 (월별 목록)
```
/schedule
   └ 월 네비 [◀] 2026년 5월 [▶]
   └ 카드 리스트 (날짜 오름차순):
        · 좌측: 날짜(큰 글씨) + 요일
        · 중앙: 제목 + 유형 라벨 + 설명 + 작성자·작성일
        · 우측: 유형 아이콘 + 수정·삭제(직원)
        · 공휴일: 날짜·제목 빨간색 + 🚩
        · 생일: 어르신 아바타 + 🎁
        · 시설 행사: 🎉 + 설명·작성자
   └ [+ 일정 등록] (직원만): 등록 모달
   └ 모달:
        · 날짜 (datepicker)
        · 유형 (라디오)
        · BIRTHDAY 선택 시 어르신 드롭다운 노출
        · 제목 (필수) / 설명 (선택, 500자)
        · [취소] [저장] / [삭제]
```

### 권한 분기
| 화면 | CAREGIVER | SOCIAL_WORKER·ADMIN | GUARDIAN |
|---|---|---|---|
| 목록 조회 | ✓ | ✓ | ✓ (전용 경로) |
| [+ 등록] 버튼 | 숨김 | 표시 | 숨김 |
| 카드 수정·삭제 | 숨김 | 표시 | 숨김 |

식단·일정 작성·수정·상세 보기 모두 **공용 모달 컴포넌트** 재사용. 4차 공지(별도 페이지)와 다른 패턴 — 입력 항목이 적고 빠른 작업이라 모달이 적합.

## 4단계. 데이터 모델 (델타)

### 신규 enum
```sql
CREATE TYPE meal_type AS ENUM ('BREAKFAST','LUNCH','DINNER','SNACK');
CREATE TYPE schedule_event_type AS ENUM ('FACILITY_EVENT','BIRTHDAY','HOLIDAY');
```

### 신규 테이블: meal_log
| 컬럼 | 타입 | 제약 | 설명 |
|---|---|---|---|
| id | bigint | PK | |
| facility_id | bigint | FK → facility | 시설 격리 |
| author_id | bigint | FK → app_user | 등록 직원 |
| meal_date | date | NOT NULL | 식사 날짜 (오늘 이하 강제, 어플리케이션 검증) |
| meal_type | meal_type | NOT NULL | |
| menu_text | text | NOT NULL | 줄단위 메뉴 |

| created_at / updated_at / deleted_at | timestamptz | | soft delete |

**UNIQUE 제약 (중요)**: 같은 facility+date+type 중복 방지
```sql

    ON meal_log (facility_id, meal_date, meal_type) WHERE deleted_at IS NULL;
CREATE INDEX idx_meal_log_facility_date
    ON meal_log (facility_id, meal_date) WHERE deleted_at IS NULL;
```

photos JSONB:
```json

```

| 컬럼 | 타입 | 제약 | 설명 |
|---|---|---|---|
| id | bigint | PK | |
| facility_id | bigint | FK → facility | |

| event_date | date | NOT NULL | |
| event_type | schedule_event_type | NOT NULL | |

```sql
    ON schedule_event (facility_id, event_date) WHERE deleted_at IS NULL;

    ON schedule_event (resident_id)
    WHERE deleted_at IS NULL AND event_type = 'BIRTHDAY';
```

### 사진 저장 인프라
- 위치: `{UPLOAD_DIR}/meals/{yyyy}/{mm}/{uuid}.{ext}` 자동 폴더 생성
- 파일명: 서버가 `{uuid4}.{ext}` 생성 (원본명 무시, 보안)
- 서빙: FastAPI 정적 마운트, URL `{STATIC_BASE_URL}/meals/{yyyy}/{mm}/{uuid}.{ext}`
- 제한: 1장당 ≤5MB, 확장자 jpg/jpeg/png/webp, **MIME 타입 검증 필수**
- 삭제: meal_log soft delete해도 사진 파일 물리 삭제 X (운영 cleanup 추후)

## 5단계. API 명세

### 신규 권한 의존성
```python
def require_content_editor(user = Depends(current_user)):
    if user.role not in ('SOCIAL_WORKER', 'ADMIN'):
        raise HTTPException(403, "FORBIDDEN")
    return user
```

### 신규 11개 + 변경 1개
| # | 메서드 | 경로 | 의존성 |
|---|---|---|---|
| 1 | GET | /api/meals | require_staff |
| 2 | POST | /api/meals | require_content_editor |
| 3 | PATCH | /api/meals/{id} | require_content_editor |
| 4 | DELETE | /api/meals/{id} | require_content_editor |
| 5 | POST | /api/uploads/meal-photo | require_content_editor |
| 6 | GET | /api/schedule | require_staff |
| 7 | POST | /api/schedule | require_content_editor |
| 8 | PATCH | /api/schedule/{id} | require_content_editor |
| 9 | DELETE | /api/schedule/{id} | require_content_editor |

| 변경 | GET | /api/home/summary, /api/parent/me | meals·schedule 카운트 추가 |

### 1. GET /api/meals?date=YYYY-MM-DD
응답:
```json
{
  "date":"2026-05-26", "weekday":"TUESDAY", "isToday":true,
  "meals":[ {"id":12,"mealType":"BREAKFAST","menuText":"...","photos":[{"url":"..."}],
             "author":{"id":2,"name":"박서준"},"createdAt":"..."} ]
}
```
빈 식사구분은 응답에 없음 — 프론트가 4종 중 빠진 것을 빈 행으로 렌더. facility_id 토큰 자동 필터. 미래 날짜 요청 시 400.

### 2. POST /api/meals
요청:
```json
{ "mealDate":"2026-05-26", "mealType":"LUNCH",
  "menuText":"잡곡밥\n미역국\n고등어구이",
  "photos":[ { "url":"/static/meals/2026/05/def.jpg" } ] }
```
**서버 검증 필수**:
- `mealDate <= 오늘` → 미래면 400 FUTURE_DATE_NOT_ALLOWED
- `photos.length <= 2`
- 각 photo.url이 `/static/meals/`로 시작하는지 (외부 URL 주입 차단)
- 같은 facility+date+type 중복 시 409 DUPLICATE_MEAL

facility_id·author_id는 토큰에서 자동.

### 3. PATCH /api/meals/{id}
`mealType`·`menuText`·`photos` 중 변경. **`mealDate` 변경 불가** (날짜 바꾸려면 삭제 후 재등록). mealType 변경 시 중복 체크 동일.

### 4. DELETE /api/meals/{id}
soft delete. 사진 파일은 그대로.

### 5. POST /api/uploads/meal-photo ⭐
**요청**: multipart/form-data, field `file`

**서버 검증 (모두 필수)**:
1. 파일 크기 ≤ `UPLOAD_MAX_SIZE_MB`
2. 확장자 ∈ `UPLOAD_ALLOWED_EXT`
3. **MIME 타입 검증** (python-magic 또는 Pillow `Image.verify()`) — 확장자만으론 위조 가능
4. 파일명은 서버가 `{uuid4}.{ext}` 생성 (원본명 무시)
5. 저장: `{UPLOAD_DIR}/meals/{yyyy}/{mm}/{uuid}.{ext}` (폴더 자동 생성)

**응답 201**: `{ "url":"/static/meals/2026/05/abc.jpg", "filename":"abc.jpg", "sizeBytes":245678 }`

**에러**: `400 INVALID_FILE_TYPE` / `INVALID_IMAGE` · `413 FILE_TOO_LARGE`

날짜 오름차순. BIRTHDAY일 때만 resident 객체 포함.

### 7. POST /api/schedule
**서버 검증**:
- eventType=BIRTHDAY → residentId 필수, 다른 유형이면 null
- BIRTHDAY일 때 residentId가 본인 시설 어르신인지 검증

에러: `400 BIRTHDAY_REQUIRES_RESIDENT` / `INVALID_RESIDENT_FOR_BIRTHDAY`

### 8-9. PATCH/DELETE /api/schedule/{id}
PATCH는 eventDate·eventType·title·description·residentId 중 변경. eventType 변경 시 위 검증 동일.

### 10-11. GET /api/parent/meals, /api/parent/schedule
파라미터 동일. **응답에서 author 필드 제외**, `/api/parent/meals`는 id·createdAt도 제외(보호자 정보 보안). 보호자에게 노출 안 함.

### AI에게 시킬 때 황금 규칙 (6차 추가)
- **6차는 AI 호출 0회 스프린트**. Gemini 부르는 코드 추가 금지.
- 사진 업로드는 **반드시 MIME 타입 검증**. 확장자만 보면 위조 가능.
